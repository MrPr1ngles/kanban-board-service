# Модель авторизации

## Общий принцип

Аутентификация построена на **JWT (JSON Web Token)**. Пользователь обменивает пару
`login + password` на подписанный токен, который передаётся в каждом защищённом запросе
в заголовке `Authorization: Bearer <token>`. Сервер проверяет подпись и срок действия
токена, извлекает идентификатор пользователя и определяет его права.

Все эндпоинты, кроме `POST /api/auth/register` и `POST /api/auth/login`, требуют
валидный токен.

## Хранение паролей

Пароли **никогда не хранятся в открытом виде**.

| Плохой способ (недопустимо) | Безопасный способ (реализовано) |
|---|---|
| `password: my_password_123` | `password_hash: $2b$12$…` (bcrypt) |

При регистрации пароль хэшируется алгоритмом **bcrypt** со встроенной солью — в базе
остаётся только хэш. При входе введённый пароль повторно хэшируется и сравнивается
с сохранённым значением функцией `bcrypt.checkpw`.

Выбор bcrypt обоснован: адаптивная функция с настраиваемой стоимостью, встроенная соль,
устойчивость к перебору на GPU. Алгоритм по заданию свободный — bcrypt является отраслевым
стандартом для этой задачи.

## Жизненный цикл токена

```
POST /auth/register {login, password}  ->  bcrypt.hash  ->  INSERT users
POST /auth/login    {login, password}  ->  verify hash  ->  JWT {sub: user_id, exp}

Любой защищённый запрос:
  Authorization: Bearer <JWT>
    -> проверка подписи (HS256) и срока действия exp
    -> current_user
    -> проверка роли на доске (RBAC, см. Роли_и_права.md)
```

Параметры токена: алгоритм подписи **HS256**, срок жизни **24 часа**, полезная нагрузка
содержит `sub` (id пользователя), `iat` (время выпуска), `exp` (время истечения).
Секрет подписи задаётся через переменную окружения `JWT_SECRET`.

## Обработка ошибок авторизации

| Ситуация | Код ответа |
|---|---|
| Нет токена / некорректный токен | 401 Unauthorized |
| Истёк срок действия токена | 401 Unauthorized |
| Недостаточно прав для действия | 403 Forbidden |
| Нет доступа к доске (скрываем факт существования) | 404 Not Found |

## Аудит действий

Каждое изменение доски (создание, правка, перемещение, удаление сущностей, изменение
состава участников) фиксируется в таблице `audit_log`: пользователь, сущность, действие,
полезная нагрузка (diff). Запись аудита выполняется **в той же транзакции**, что и основная
операция, поэтому журнал всегда согласован с данными. История доски доступна владельцу
через `GET /api/boards/{id}/audit`.

## Реализация в коде

- `app/core/security.py` — `hash_password`, `verify_password` (bcrypt), `create_token`, `decode_token` (JWT).
- `app/core/permissions.py` — `get_current_user` (извлечение пользователя из токена), `require_board` (проверка роли).
- `app/services/audit_service.py` — запись в журнал изменений.
