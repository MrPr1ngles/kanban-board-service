# Роли и права доступа

## Выбранная модель: RBAC

Применяется **RBAC (Role-Based Access Control)** — пользователю на уровне доски
назначается одна из трёх ролей. Роли образуют иерархию по правам:
`reader ⊂ writer ⊂ owner` (каждая следующая включает права предыдущей).

Реализован **принцип минимальных привилегий**: доска, к которой у пользователя нет
доступа, неотличима от несуществующей — сервер отвечает `404 Not Found`, не раскрывая
сам факт её наличия.

## Таблица прав (минимум 3 роли)

| Роль | Доступ к чтению | Доступ к редактированию | Доступ к администрированию | Управление ролями |
|---|:---:|:---:|:---:|:---:|
| **Reader** (Читатель) | Да | Нет | Нет | Нет |
| **Writer** (Писатель) | Да | Да | Нет | Нет |
| **Owner** (Владелец) | Да | Да | Да | Да |

**Расшифровка:**
- *Чтение* — просмотр доски, колонок, карточек, комментариев, участников.
- *Редактирование* — создание/правка/удаление колонок и карточек, перемещение, комментарии.
- *Администрирование* — удаление доски, просмотр журнала аудита.
- *Управление ролями* — добавление/исключение участников, изменение их ролей.

## Сопоставление ролей и эндпоинтов

| Действие | Минимальная роль |
|---|---|
| Просмотр доски, колонок, карточек, комментариев | reader |
| Создание/правка/удаление колонок и карточек | writer |
| Перемещение карточек (drag-and-drop) | writer |
| Добавление комментариев | writer |
| Удаление доски | owner |
| Управление участниками и ролями | owner |
| Просмотр журнала аудита | owner |

## Обоснование выбора: RBAC против ABAC

**ABAC (Attribute-Based Access Control)** накладывает доступ на комбинацию атрибутов
пользователя и ресурса. Он оправдан там, где правила зависят от множества динамических
факторов: время суток, подразделение сотрудника, гриф секретности, геолокация и т. п.

В домене Kanban-доски **таких атрибутивных условий нет**: полный набор прав описывается
тремя фиксированными ролями, привязанными к паре «пользователь — доска». Поэтому:

- **RBAC даёт необходимую выразительность** — все требуемые сценарии доступа покрываются.
- **RBAC значительно проще** в реализации и сопровождении: одна проверка `роль ≥ требуемой`.
- **ABAC был бы избыточным усложнением** без практической пользы для данной задачи.

Вывод: для Kanban-доски выбран RBAC как оптимальный по соотношению «выразительность/сложность».

## Реализация в коде

Роли хранятся в таблице `board_members (board_id, user_id, role)`. Проверка выполняется
зависимостью `require_board(db, board_id, user, min_role)` в `app/core/permissions.py`,
которая сверяет роль пользователя с требуемой по иерархии `ROLE_ORDER`.
